### 0.1 СДЕЛАЛ ОДНО ИЗМЕНЕНИЕ → НЕМЕДЛЕННО ТЕСТ

```
1 изменение = 1 цикл:
  написать код → build (timeout 300с) → serve (timeout 30с) → e2e тест (timeout 90с) → скриншот → Vision
```

**НЕ наваливать 5 изменений подряд без теста.** Одно изменение → один тест → потом следующее.
Если сделал 3 правки без теста — ты НАРУШИЛ правило. Вернись и протестируй.

### 0.2 ЛЮБАЯ КОМАНДА — С ТАЙМАУТОМ

| Команда | Таймаут | Что делать при таймауте |
|---------|---------|------------------------|
| `build.ps1` | 300с | Kill Godot, сообщить ошибку |
| `http-server` | 30с подъём | Kill, перезапустить |
| `jest` | 90с/тест | Записать как FAIL, чинить |
| `mcp-cli` (Vision) | 90с | Kill, fallback |

**Команда без таймаута = потенциальный вечный вис на 3 часа.**
ВСЕГДА передавай `-TimeoutSec` в bash-вызовах.

### 0.3 НЕ ГОВОРИ «РАБОТАЕТ» БЕЗ ДОКАЗАТЕЛЬСТВА

- Скриншот + Vision-MCP анализ = единственное доказательство.
- Тест зелёный = единственное доказательство.
- «Должно работать» = НЕ работает. Докажи.

### 0.8 ВЕРИФИКАЦИЯ ДЕПЛОЯ В РЕАЛЬНОМ БРАУЗЕРЕ (Puppeteer + Vision)

> ⚠️ Добавлено 2026-06-24 после кейса «чёрный экран»: фронтенд задеплоили, файлы на сервере,
> curl отдаёт 200 — а игрок видит чёрный экран. Причина: дефолтный Godot HTML-shell не показывает
> прогресс, и пока качаются 112 МБ, экран тёмный. **curl/HEAD НЕ доказывают, что игра работает.**

**ПРАВИЛО: после ЛЮБОГО деплоя (билд на сервер, фронтенд, сцена) — НЕ говорить «работает», пока не проверено в реальном браузере через Puppeteer + Vision-MCP.** curl `200` и «файлы залиты» = НЕ доказательство.

**Обязательный цикл проверки деплоя:**
1. **Puppeteer грузит live-URL** (или локальный `http-server :3000`, если канал агента до сервера плохой — см. ниже) с throttled-сетью (`Network.emulateNetworkConditions`, ~1 МБ/с), чтобы поймать экран загрузки.
2. **Скриншот экрана загрузки** → Vision-MCP (`analyze_image`): подтвердить, что виден брендинг/прогресс/%, а НЕ чёрный/пустой экран.
3. **Скриншот после загрузки** (меню/игра) → Vision-MCP: подтвердить, что кириллица рендерится, кнопки видны, цвета верные.
4. **Полный e2e** (минимум `regression_smoke` + загрузка + меню) → 0 падений.

**Если канал агента до сервера не тянет 112 МБ (throttle/loss — было у самого автора):**
- Воспроизвести **локально**: `npx http-server builds/html5 -p 3000` (localhost, без throttle) → тот же Puppeteer-цикл по `http://localhost:3000/`. Билд идентичен — баг воспроизводится.
- На сервере отдельно проверить только транспорт: HTTP `200` + **content-type** для `/. /index.js (application/javascript), /index.wasm (application/wasm), /index.pck (octet-stream)` + наличие `/ws` прокси.
- НЕ закрывать задачу «работает у пользователя» без локальной Puppeteer+Vision проверки того же билда.

**Для Godot web-билдов — ВСЕГДА:**
- Применять кастомный shell (`scripts/dev/patch_web_shell.js` после `build.ps1`) с экраном загрузки (BOOKWAR + прогресс % + watchdog 45с). Дефолтный shell = чёрный экран во время загрузки = репорт «сломано».
- Проверить `Engine.getMissingFeatures()` не падает (иначе оверлей ошибки вместо игры).

### 0.9 ЛЮБОЕ ИЗМЕНЕНИЕ → И ДЛЯ ТЕЛЕФОНОВ, И ДЛЯ КОМПОВ (ЗАЯВЛЕНО ПОЛЬЗОВАТЕЛЕМ 2026-06-25)

> ⚠️ Это ГЛОБАЛЬНОЕ правило. Применяется ко ВСЕМ фичам/фиксам/правкам UI/геймплея.

**ПРАВИЛО:** Любая просьба пользователя «сделай X» означает: **X должно работать и на mobile (touch), и на desktop (mouse + keyboard)**. Не «сделать только для одной платформы» — обе сразу.

**Что это значит на практике (ОБЯЗАТЕЛЬНО для каждой UI/геймплей-правки):**

1. **Touch handler наравне с mouse/keyboard.** Любой новый Control, кнопка, зона тапа —
   обрабатывать `InputEventScreenTouch` так же надёжно, как `InputEventMouseButton` и
   `InputEventKey`. В Godot HTML5 `Input.emulate_mouse_from_touch` **ненадёжен**
   (проверено 2026-06-25: ловит только верхние кнопки + даёт двойную обработку → toggle cancels).
   **Решение:** кастомный `_input(event)` с явным hit-test по `btn.get_global_rect().has_point(pos)`
   и **прямым вызовом** обработчика (не через `emit_signal("pressed")` — signal-connection может
   не отработать если _ready упал на дочерней сцене). См. эталон `scripts/ui/main_menu.gd`.

2. **Координаты touch маппятся через АКТУАЛЬНЫЙ viewport.** При `stretch/aspect="expand"`
   Godot меняет viewport под aspect дисплея (например, 1280×720 → 1676×720 при wide screen).
   `Control.get_global_rect()` возвращает coords в АКТУАЛЬНОМ viewport, не в базовом 1280×720.
   ВСЕ тесты/Puppeteer должны читать `window.gameViewportSize` (или аналог) и маппить
   CSS → Godot через `css * (viewport / canvasBox)`. Хардкод `1280×720` = баг.

3. **HTML5 AudioContext unlock.** На web музыка/звук заблокированы до первого user gesture.
   Каждое новое место где может играть звук — убедиться что первый tap/click вызывает
   `audioContext.resume()` (см. `patch_web_shell.js` — обработчик на `touchstart`/`mousedown`).
   Без этого «музыка не играет» на 100% случаев.

4. **Browser-only кнопки прячутся или заменяются.** `window.close()`, `get_tree().quit()`,
   доступ к файловой системе — браузеры блокируют. Если фича требует этого — скрыть на web
   через `if OS.has_feature("web"): btn.visible = false` (как кнопка Quit в `main_menu.gd`).

5. **Desktop-only фичи работают через keyboard/mouse.** Если фича требует клавиатуры
   (WASD, hotkeys) — оставить, но убедиться что есть и touch-альтернатива (виртуальный
   D-pad, кнопки действий). Сцены мира/боя имеют touch-buttons в HUD — проверять их.

**ПРИ ПРОВЕРКЕ (GATE 2):** Puppeteer-тест прогонять в **ДВУХ режимах**:
- `setViewport({ isMobile: true, hasTouch: true })` + `page.touchscreen.tap(x,y)` — mobile
- `setViewport({ isMobile: false })` + `page.mouse.click()` + `page.keyboard.press()` — desktop

Если работает только в одном — **задача НЕ завершена**, чинить до обоих.

**АНТИ-ПАТТЕРН (ЗАПРЕЩЕНО):**
- Сказать «сделано» после проверки только на desktop (мышью/клавиатурой)
- Полагаться на `Input.emulate_mouse_from_touch` как на единственный touch-путь
- Использовать хардкод `1280×720` для координат (viewport меняется при expand)
- Скрыть фичу «потому что на мобильном сложно» (правильно — сделать touch-альтернативу)
- Обрабатывать `emit_signal("pressed")` без прямого вызова обработчика (signal-connect ненадёжен)

**ЭТАЛОНная реализация:** `scripts/ui/main_menu.gd` — кастомный `_input` + `_dispatch_button` +
`window.gameButtonRects` + `window.gameViewportSize` + debounce 300мс (от двойных touch+emulate).

### 0.11 ПРИНЦИП «1 ДЕЙСТВИЕ = 1 ЭФФЕКТ» (ЗАЯВЛЕНО ПОЛЬЗОВАТЕЛЕМ 2026-06-25)

> ⚠️ Корневая причина большинства UI-багов в BOOKWAR (легенда перескакивает 1→3,
> toggle cancels, кнопки не реагируют) — **двойная/тройная обработка одного тапа**.
> Этот принцип запрещает такие баги системно.

**ПРАВИЛО: на HTML5 web-export КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО `Button.pressed.connect(handler)`
для интерактивных UI-кнопок (меню, легенда, инвентарь, крафт, multiplayer, бой).**

**Причина:** Godot 4.6 HTML5 на один физический tap генерирует сразу НЕСКОЛЬКО input events:
1. `InputEventScreenTouch` (нативный touch)
2. `InputEventMouseButton` (через `emulate_mouse_from_touch`)
3. Для каждого — отдельный проход `_gui_input` → `pressed` signal

Если обработчик подключён и через `pressed.connect`, и через кастомный `_input` hit-test —
**один тап = 2-3 вызова handler'а**. Для toggle-логики (открыть/закрыть) = cancel. Для
advance-логики (листать страницы) = перескок через 1-2 страницы. Для craft — расход
модификаторов вдвое быстрее.

**ЕДИНСТВЕННЫЙ разрешённый паттерн (mobile + desktop):**

```gdscript
extends Control

var _last_action_time: float = 0.0  # debounce guard

func _input(event: InputEvent) -> void:
    # Accept BOTH touch AND mouse — universal handler.
    var pos := Vector2(-1, -1)
    var is_press := false
    if event is InputEventScreenTouch:
        pos = event.position
        is_press = event.pressed
    elif event is InputEventMouseButton and event.pressed and event.button_index == MOUSE_BUTTON_LEFT:
        pos = event.position
        is_press = true
    if not is_press or pos.x < 0:
        return
    # Debounce 200-300ms — kill double-firing from touch+emulate.
    var now: float = Time.get_ticks_msec() / 1000.0
    if now - _last_action_time < 0.25:
        return
    # Hit-test all interactive buttons, dispatch the FIRST match.
    if _button_a and _button_a.get_global_rect().has_point(pos):
        _last_action_time = now
        _on_button_a_pressed()
        get_viewport().set_input_as_handled()
        return
    if _button_b and _button_b.get_global_rect().has_point(pos):
        _last_action_time = now
        _on_button_b_pressed()
        get_viewport().set_input_as_handled()
        return

func _on_button_a_pressed() -> void:
    # Direct handler — NO emit_signal("pressed"), NO pressed.connect elsewhere.
    pass
```

**ОБЯЗАТЕЛЬНЫЕ правила:**

1. **`Input.emulate_mouse_from_touch = false`** в каждом `_ready()` интерактивной сцены.
   Touch приходит только как `InputEventScreenTouch` (один event, не два).
2. **Никаких `pressed.connect(handler)`** для кнопок которые что-то делают (смена сцены,
   toggle, advance страниц, выбор буквы). Только `_input` hit-test → прямой вызов.
3. **Debounce ≥ 200мс** на каждый `_input` — защита от оставшихся дублей.
4. **`set_input_as_handled()`** после срабатывания — блокирует downstream processing.
5. **Hit-test по `get_global_rect().has_point(pos)`** — НЕ по хардкод-координатам.
6. **Прямой вызов обработчика** (`_on_button_xxx_pressed()`), НЕ `emit_signal("pressed")`.

**ИСКЛЮЧЕНИЯ (где `pressed.connect` ещё допустимо):**
- НеИнтерактивные подписки на внешние сигналы (`InventoryManager.inventory_changed.connect(_refresh)`).
- Hold-кнопки D-pad в HUD (`button_down`/`button_up`) — там логика press+release, не toggle.
- Сигналы `gui_input.connect(_on_cell_input)` для клеточных grid'ов — НО только при условии
  что внутри обработчика есть тот же debounce.

**ТРЕБОВАНИЕ К ТЕСТАМ (GATE 2 для UI):**

Каждая новая UI-кнопка ОБЯЗАНА сопровождаться regression-тестом:

```javascript
// В tests/e2e — обязательный чек: 1 tap = 1 эффект, не больше.
test('intro panel advance: 1 tap → +1 panel', async () => {
  const before = await page.evaluate(() => window.gameIntroIndex);
  await page.touchscreen.tap(nextBtn.cx, nextBtn.cy);
  await sleep(500);
  const after = await page.evaluate(() => window.gameIntroIndex);
  expect(after).toBe(before + 1);  // ← НЕ +2, НЕ +3
});
```

Тот же тест прогоняется и в mobile-режиме (`hasTouch:true, page.touchscreen.tap`)
и в desktop (`page.mouse.click`). Если в любом из них `delta != 1` — **баг, фиксить**.

**АНТИ-ПАТТЕРН (ЗАПРЕЩЕНО):**
- `btn.pressed.connect(handler)` + `_input` hit-test на ту же кнопку
- `emit_signal("pressed")` из кастомного handler'а (вызывает signal subscribers + наш код)
- Забывать `set_input_as_handled()` — downstream nodes тоже обработают event
- Debounce < 200мс (Touch + emulate-mouse приходят с разницей ~50-100мс)
- Тестировать UI только на desktop (touch duplicate проявляется только на mobile)

**ЭТАЛОНЫ:** `scripts/ui/main_menu.gd`, `scripts/ui/intro.gd`, `scripts/ui/inventory_ui.gd`,
`scripts/ui/multiplayer_ui.gd` — все кнопки через `_input` + hit-test + debounce.

3. **Patch shell** → `node scripts/dev/patch_web_shell.js` (build перегенерирует index.html,
   патч нужно применять **после** каждого build).
4. **Локальный serve** → `http-server builds/html5 -p 3000` (один раз, держать запущенным).
5. **Puppeteer-тест локально** → `node <test>.js` против `http://localhost:3000/`.
   - Канал локальный = мгновенная загрузка 112МБ из памяти (1-3с).
   - Все манипуляции (tap, keyboard, мосты `window.game*`) работают.
6. **Vision-MCP** на ключевые скриншоты → убедиться что кириллица, цвета, кнопки ОК.
7. **e2e regression** → `cd tests/e2e && npx jest --testPathIgnorePatterns capture.test.js` → 0 падений.
8. **ТОЛЬКО когда всё зелёное локально → деплой на сервер** через `scp -C -i <key> ... root@<bookwar-server-ip>:/var/www/bookwar/`.
9. **После деплоя** — `curl` transport-проверка (content-type + размер + наличие патча в index.html).
   Полный Puppeteer-test против live — **только если канал до сервера позволяет скачать 112МБ**
   (curl --max-time 300 если зависает → канал не годится, проверка transport-только).
10. **Сказать «работает на сервере» можно только после шага 9.** До этого — «работает локально, деплой подтверждён transport-проверкой».

**ПОЧЕМУ ТАК:** канал агента до сервера <bookwar-server-ip> через VPN нестабилен — `curl index.wasm`
может висеть 6+ минут и обрываться с `ERR_CONNECTION_RESET`. Chrome через Puppeteer тогда падает
с `Engine is not defined` и рендерит чёрный экран. Это **не баг игры** — это ограничение сети агента.
Игроки с нормальным интернетом (как подтверждено пользователем на Android-телефоне) грузят без проблем.

**АНТИ-ПАТТЕРН (ЗАПРЕЩЕНО):**
- Деплоить на сервер, не протестировав локально
- Говорить «работает на сервере» без шагов 8-9
- Делать iteration-цикл «правка → деплой → live-test» для каждой мелочи (это часы на каждый цикл)
- Пропускать `patch_web_shell.js` после build (дефолтный shell = чёрный экран при загрузке)

## 2. БУКВЕННАЯ СИСТЕМА — ЯДРО ИГРЫ

### 2.0 ПРИНЦИП АЛФАВИТ-ЗАВИСИМОСТИ ЛОКАЛИ (ГЛОБАЛЬНЫЙ)

> ⚠️ ФУНДАМЕНТАЛЬНЫЙ дизайн-принцип. Вся игровая структура ПРОИЗВОДНА от алфавита
> выбранной локали. Русский = 33 буквы → 33 уровня и 33 «оружия/брони». У других
> языков число букв ДРУГОЕ → **меняется число уровней, число предметов и баланс**.
> Локализация = НЕ перевод строк, а ПЕРЕСТРОЕНИЕ игры под алфавит локали.

**Жёсткая связка «алфавит → структура игры»:**

| Параметр игры | Значение | Зависит от |
|---------------|----------|------------|
| Число уровней | = N | N = длина алфавита локали |
| Число «оружия/брони» (букв-предметов) | = N | N = длина алфавита |
| Уровень буквы / сила / скорость | позиция в алфавите | не меняется (универсально) |
| Баланс атаки/защиты | соотношение гласных/согласных | РАЗНОЕ per-локаль |
| Финал (последний уровень) | уровень N | босс + самые сильные/дорогие буквы |

**Примеры по локалям** (N = длина алфавита → N уровней):

| Локаль | Букв (N) | Уровней | Гласные / Согласные | Баланс |
|--------|----------|---------|---------------------|--------|
| ru | 33 | 33 | 10 / 21 + 2 знака | текущая версия |
| en | 26 | 26 | 5 / 21 | defense-heavy → множитель гласных ↑ |
| es | 27 | 27 | 5 / 22 | +ñ |
| de | 30 | 30 | varies | +ä,ö,ü,ß |
| ar | 28 | 28 | 3 краткие+3 долгие / остальное | RTL |
| zh | ??? | ??? | нет алфавита | спец-дизайн (см. ниже) |

**Архитектурное следствие (для агента):**
- НИЧЕГО не хардкодить «33». `EXPECTED_LETTER_COUNT`, длина `MAP_CHAIN`, `levels.json`
  должны быть **производными** от загруженного `data/letters_{locale}.json`.
- На старте: `I18n.get_locale()` → грузит алфавит локали → от его длины N
  **генерируются**: число уровней, прогрессия (уровень N/2 = середина), спавнер,
  список букв-предметов, число заклинаний.
- Множители боя (`VOWEL_MULTIPLIER`, `CONSONANT_MULTIPLIER`) — **per-locale**, т.к.
  соотношение гласных/согласных разное (en: 5/21 → дефолт defense-heavy; ru: 10/21).
- Заклинания (слова из букв) — **свои** per-локаль (из букв этого алфавита).
- Прогрессия сложности/насыщенности: масштабируется к N, уровень N/2 = середина
  (см. §0.12 cleanup и формулу буквиц в `world_map._spawn_generic_items`).

**Китайский (zh) — особый случай:** у китайского НЕТ фиксированного алфавита
(тысячи иероглифов), модель «1 буква = 1 уровень» ломается. Варианты:
(a) пиньинь (26 латинских) — как en; (b) N ключевых иероглифов как «алфавит»;
(c) 214 ключей Канси → 214 уровней (тогда zh = самая длинная версия игры).
Решение зафиксировать в `data/letters_zh.json`. См. TODO.md → I18N-6.

**АНТИ-ПАТТЕРН (ЗАПРЕЩЕНО):**
- Хардкодить `33` где-либо вне ru-специфичных данных.
- Считать, что локаль = только перевод строк (она перестраивает ВСЮ игру).
- Применять один баланс для всех локалей (отношение vowel/consonant разное).

### 2.2 Уровень буквы = количество копий

У игрока нет «уровня» в привычном смысле. У каждой буквы есть свой уровень.

- **Уровень буквы = количество собранных копий этой буквы**
- 1 копия А = уровень 1. Урон = 33 × 1 = 33
- 3 копии А = уровень 3. Урон = 33 × 3 = 99
- 5 копий Щ = уровень 5. Защита = 7 × 5 = 35

**Продажа/обмен**: Игрок может отдать 1 копию буквы другому игроку или NPC. При этом его уровень этой буквы уменьшается на 1, а у получателя — увеличивается на 1.

**Использование в бою**: При атаке/защите буква НЕ теряется. Уровень буквы — это перманентная характеристика. Но если продал копию — сила падает.

### 2.3 Формулы боя

```
base_power = 34 - speed_position    (А=33, Я=1)
speed = speed_position              (А=1, Я=33)

Для ГЛАСНЫХ (атака):
  damage = base_power × letter_level × VOWEL_MULTIPLIER
  
Для СОГЛАСНЫХ (защита):
  shield = base_power × letter_level × CONSONANT_MULTIPLIER
  
Для Ъ (твёрдый знак — бафф защиты):
  defense_bonus = base_power × letter_level × SIGN_MULTIPLIER
  (усиливает следующую согласную защиту)
  
Для Ь (мягкий знак — бафф атаки):
  attack_bonus = base_power × letter_level × SIGN_MULTIPLIER
  (усиливает следующую гласную атаку)
```

**Балансировочные константы** (настраиваются):
- `VOWEL_MULTIPLIER = 1.0` — множитель урона гласных
- `CONSONANT_MULTIPLIER = 1.0` — множитель защиты согласных
- `SIGN_MULTIPLIER = 1.5` — множитель для Ъ и Ь (они редкие, поэтому сильнее)

**Порядок хода**: В бою все участники выкладывают по 1 букве. Ходы разрешаются по скорости: от самой быстрой (33) к самой медленной (1). Быстрые бьют первыми, медленные — последними, но сильнее.

> **Правило «одна буква — один раз ЗА ВЕСЬ БОЙ»** (актуально с 2026-06-16): в пределах одного боя
> каждую букву (и каждое заклинание) можно применить ТОЛЬКО ОДИН РАЗ. За один ход можно сыграть
> несколько РАЗНЫХ букв, но использованные «сгорают» до конца боя. В новом ходе доступны лишь ещё
> не сыгранные буквы. Новый бой (новая сцена `battle_scene` → `start_combat`) полностью сбрасывает
> использование. Это создаёт ресурс-менеджмент и тактику: какие буквы тратить сейчас, какие беречь.
> Реализовано в `combat_system.gd` через `_player_played_this_battle` / `_player_spells_this_battle`
> (сбрасываются ТОЛЬКО в `start_combat`; НЕ сбрасываются в `_end_turn_round`).
> UI (`battle_manager.gd`): кнопки использованных букв остаются disabled и серыми до конца боя
> (`_build_hand_buttons` проверяет `_combat_system.is_letter_played(...)` каждый ход).
> Автобой (`_run_auto_battle_turn`) выбирает ещё не сыгранную букву.
> **Уровень букв постоянен** — продажа/крафт меняют уровень, но бой его не снижает.

**HP**: У каждого участника боя есть HP. Когда HP ≤ 0 — бой проигран.

### 2.4 Стартовое состояние игрока

Игрок начинает игру **без единой буквы**. Вообще пустой инвентарь.

На старте можно найти только:
- Точки `.` (ресурс, лежат на земле в Светлой Долине)
- Первые буквы (спрятанные, как награды за исследование)

## 3. ЗНАКИ ПРЕПИНАНИЯ — ОСОБЫЕ КАРТЫ

Знаки препинания — это особые способности, не привязанные к алфавитному порядку.

| Знак | Название | Эффект | Где найти |
|------|----------|--------|-----------|
| `.` | Точка | Базовый ресурс. Собираешь 3 точки → комбинируются в `...` | Повсюду в Светлой Долине, реже дальше |
| `...` | Многоточие | **Разговор с монстрами/NPC**. Позволяет избежать боя, получить информацию, торговать, убеждать. Ключевая механика старта. | Комбинируется из 3 точек |
| `,` | Запятая | **Пауза** — замедляет врага на 1 ход (его скорость −5) | Леса, болота |
| `?` | Вопрос | **Подозревает** — монстр использует: патрулирует, замечает движение игрока. Стартовый монстр | Только монстры (не игрок, поначалу) |
| `!` | Восклицание | **Заметил** — монстр переходит в агрессию, ускоряется, атакует. Стартовый монстр | Только монстры (не игрок, поначалу) |
| `-` | Дефис | **Уклонение/рывок** — позволяет уклониться от одной атаки | Катакомбы |
| `:` | Двоеточие | **Раскрытие** — показывает скрытые карты/намерения врага на следующий ход | Цитадели |
| `;` | Точка с запятой | **Комбо-связка** — позволяет сыграть 2 буквы за 1 ход вместо одной | Редкий, руины |
| `""` | Кавычки | **Отражение** — возвращает следующую вражескую атаку обратно | Очень редкий |
| `—` | Тире | **Прорыв** — пробивает вражескую защиту, игнорирует щит | Боссы |
| `()` | Скобки | **Ловушка** — ставится на поле, срабатывает при атаке врага (урон при атаке на тебя) | Болота |
| `...` + буква | Убеждение | Комбо: многоточие + конкретная буква = убеждение монстра определённым «словом» (расширенная дипломатия) | Продвинутая механика |

**Важно**: На старте игрок может ТОЛЬКО собирать точки и использовать многоточие. Никакого оружия, никаких атак. Единственный способ взаимодействовать с монстрами — говорить через `...`.

## 4. КАРТА МИРА

### 4.1 Общая структура

- **Единая огромная карта** без разделения на уровни/комнаты
- **33 региона** на карте (по числу букв, но регионы НЕ привязаны к конкретным буквам — в каждом регионе можно найти разные буквы)
- Реализуется через **TileMap / TileMapLayer** в Godot
- Seamless — единая карта, камера следует за игроком

### 4.2 Визуальный стиль

## 5. СИСТЕМА МОНЫСТРОВ

### 5.1 Стартовые монстры (Светлая Долина)

**Монстр «?» (Вопросительный)**:
- Патрулирует территорию
- Активность: «подозревает» — если игрок рядом, начинает искать
- Радиус обнаружения: малый
- При обнаружении → переходит в состояние «!»
- HP: низкий, но без оружия убить нельзя
- Убеждение через `...`: «Ты не знаешь, кто я. Уходи.» → монстр уходит

**Монстр «!» (Восклицательный)**:
- Агрессивный, атакует при контакте
- Быстрый, но слабый
- HP: средний
- Без оружия — только убеждение через `...`
- Убеждение: «Подожди! Я не враг!» → монстр замедляется, можно убежать

### 5.2 Прогрессия монстров

По мере продвижения вглубь карты:
- Монстры начинают использовать буквы (атакуют гласными, защищаются согласными)
- Появляются монстры с комбинациями: `?` + буква = хитрый монстр, `!` + буква = агрессивный
- Боссы регионов: уникальные монстры с набором букв и знаков
- **Финальный босс**: Хранитель Запрета — владеет полным алфавитом

### 5.3 Дроп с монстров

- После убеждения (`...`): монстр может отдать букву, точку, информацию
- После победы в бою: монстр роняет 1-3 буквы из своего набора
- Редкие монстры: роняют редкие знаки препинания

## 6. СЮЖЕТ И НАРРАТИВ

### 6.1 Лор

Когда-то в мире звучала речь. Люди писали книги, называли вещи по имени, пели песни. Буквы жили в каждом доме, в каждом слове.

Но пришёл **Хранитель Запрета** — древняя сущность, рождённая из страха перед силой слова. Он наложил проклятье: буквы были разбросаны по тёмным землям, за пределы Светлой Долины. Способность говорить и писать была утрачена. Даже в Светлой Долине — последнем оплоте — буквы запрещены под страхом изгнания.

Но нашлись герои. Те, кто решил отправиться в тёмные земли и вернуть буквы одну за одной. Восстановить алфавит. Разрушить проклятье.

### 6.2 Ключевые персонажи/NPC

- **Староста деревни** (Светлая Долина) — даёт первые задания, учит механикам
- **Тихий Странник** — NPC в лесу, обменивает точки на подсказки
- **Библиотекарь Призрак** — в руинах библиотеки, знает где спрятаны буквы
- **Кузнец Слов** — в цитадели, может улучшать (комбинировать) буквы
- **Хранитель Запрета** — финальный босс, владеет всеми 33 буквами

## 7. МУЛЬТИПЛЕЕР

### 7.1 Архитектура

- **Godot SceneMultiplayer** (высокоуровневый API)
- **ENet** для PC нативных клиентов
- **WebSocket** для web-клиентов (itch.io не поддерживает UDP/ENet)
- **Сервер**: выделенный Godot headless сервер или peer-to-peer с хостом
- **WebRTC** для голосового чата (отдельный сигнальный сервер)

### 7.2 Функционал

- **Лобби**: Создать/найти комнату
- **Текстовый чат**: глобальный + локальный (в регионе)
- **Голосовой чат**: Push-to-talk или активация по голосу, внутри группы
- **Торговля**:直接的 обмен буквами между игроками (торговое окно)
- **PvP**: Дуэли на буквах (арена или в мире)
- **Кооп**: Совместное исследование мира, групповые бои с боссами
- **Синхронизация мира**: Позиции игроков, состояние монстров, ресурсы

### 7.3 Приоритет реализации

Мультиплеер реализуется ПОСЛЕ основной одиночной игры. Сначала работающая одиночка, потом сетка.

## 10. КОНВЕНЦИИ КОДА

### 10.1 Именование

- **Файлы**: `snake_case.gd` (например, `combat_system.gd`)
- **Классы/ноды**: `PascalCase` (например, `CombatSystem`)
- **Переменные**: `snake_case` (например, `base_power`)
- **Константы**: `UPPER_SNAKE_CASE` (например, `VOWEL_MULTIPLIER`)
- **Функции**: `snake_case` (например, `calculate_damage()`)

### 10.2 Структура GDScript-файлов

```gdscript
extends Node
class_name ClassName

# Константы
const EXAMPLE_CONST = 10

# Экспортируемые переменные
@export var example_export: int = 5

# Переменные
var example_var: int = 0

# Сигналы
signal example_signal(value: int)

# Встроенные функции (_ready, _process и т.д.)
func _ready() -> void:
    pass

# Публичные функции
func public_function() -> void:
    pass

# Приватные функции
func _private_helper() -> void:
    pass
```

### 10.3 Правила

- Типизировать всё явно (`var x: int = 0`, не `var x = 0`)
- Не использовать `@onready` для Autoload-синглтонов (они уже готовы)
- Предпочитать Composition over Inheritance
- Комментарии — только для сложной логики, не для очевидного

> **Puppeteer (живой ввод) + скриншот + Vision-MCP анализ**.

**GATE 1 — Перед любым изменением:**
1. Прочитай AGENTS.md §2 (формулы), §12.4 (этот протокол), §13 (тесты)
2. Убедись что понял задачу — если сомневаешься, спроси

**GATE 2 — После каждого изменения (цикл фикс-верификация):**

> **ПРИНЦИП «СДЕЛАЛ → ПРОВЕРИЛ → ПРОДОЛЖИЛ»**: каждое изменение проходит полный цикл
> верификации ДО перехода к следующей задаче. Не堆ить изменения — одно за другим,
> каждое подтверждено. Тест провалился = чини ЭТО, потом двигайся дальше.
>
> ⚠️ **ЗАКОН: любое изменение кода/сцены → НЕМЕДЛЕННО e2e тест.** Не «потом», не «в конце» —
> прямо сейчас. Build → serve → jest → screenshot → Vision. Если не тестировал = не сделал.

```
foreach изменение (по одной задаче за раз):
  1. СДЕЛАЛ — написал код/сцену/ассет
  2. ПРОВЕРИЛ:
     a. build          → & scripts/dev/build.ps1 -TimeoutSec 300  (exit≠0 = стоп)
     b. kill old procs → Get-NetTCPConnection :3000 | Stop-Process
     c. serve          → Start-Job http-server :3000  (таймаут подъёма 30с)
     d. e2e test       → npx jest <затронутый suite> --testTimeout 90000
     e. screenshot     → Puppeteer: godot.takeScreenshot('метка')
     f. Vision analyze → mcp-cli call zai-vision analyze_image → прочитай ответ
  3. ОК? → коммит + следующая задача
     НЕ ОК? → чини → goto 1 (не двигайся дальше пока это не работает)
  4. cleanup          → Stop-Job + kill :3000 (ВСЕГДА)
```

**GATE 3 — Перед словами «готово» (полная верификация):**
1. **Скриншот КАЖДОГО экрана**: меню, мир, инвентарь, диалог, бой, бой-после-хода
2. **Vision-MCP анализ КАЖДОГО скриншота** — подтвердить: кириллица рендерится, кнопки видны, цвета верные, нет розовых/пустых экранов
3. **walkthrough.test.js** — бот проходит регион от старта до победы
4. **Полный regression-сьют**: `npx jest --testPathIgnorePatterns capture.test.js` → 0 падений
5. **Зачистка**: убить все серверы/браузеры, освободить порт 3000

**НИКОГДА:**
- НЕ запускай новый сервер, не убив старый (копятся процессы)
- НЕ говори «работает» без скриншота+Vision
- НЕ вызывай внутренние GDScript-функции из тестов (только Puppeteer+window.*)
- НЕ оставляй висящие процессы после команды (таймаут + cleanup ВСЕГДА)
- НЕ пропускай GATE 3 «потому что мелкое изменение» — мелкое ломает чаще крупного

### 12.5 Рабочий цикл AI-агента (SELF-DRIVEN DEVELOPMENT)

Агент работает **самостоятельно**, видит игру глазами (через Vision MCP) и итерирует без человека:

1. **Код** → написать/изменить
2. **Build** → `build.ps1` (таймаут!)
3. **Serve** → kill старого → старт нового на :3000 (таймаут подъёма!)
4. **Скриншоты** → Puppeteer делает скрин КАЖДОГО затронутого экрана
5. **Vision MCP** → `mcp-cli call zai-vision analyze_image` → прочитать описание → найти баги
6. **Фикс** → если Vision видит проблему → чини → goto 2
7. **E2E тесты** → `npx jest` → 0 падений → если есть → чини → goto 2
8. **Walkthrough** → `walkthrough.test.js` → регион проходится от старта до победы
9. **Cleanup** → убить сервер, освободить порт, закрыть браузеры

**Агент НЕ имеет права сказать «готово» пока GATE 3 полностью пройден.**

### 12.6 Vision MCP Server — глаза AI-агента

Vision-анализ доступен **ДВУМЯ способами** (использовать оба, fallback друг на друга):

#### Способ 1 — встроенные инструменты opencode (in-process)

В `opencode.json` подключён `zai-mcp-server` с `Z_AI_API_KEY`. Доступно напрямую как вызовы функций:
- `zai-mcp-server_analyze_image` — общий анализ изображения
- `zai-mcp-server_extract_text_from_screenshot` — OCR (распознавание текста)
- `zai-mcp-server_diagnose_error_screenshot` — диагностика ошибок
- `zai-mcp-server_ui_diff_check` — сравнение скриншотов (до/после)
- `zai-mcp-server_understand_technical_diagram` — анализ диаграмм
- `zai-mcp-server_analyze_data_visualization` — анализ графиков
- `zai-mcp-server_ui_to_artifact` — UI → код/спецификация
- `zai-mcp-server_analyze_video` — анализ видео (MP4/MOV/M4V, ≤8MB)

Параметры: `image_source` (локальный путь или URL) + `prompt`.

#### Способ 2 — внешний `mcp-cli.exe` (fallback, работает из любого проекта)

- Бинарник: `~\.bun\bin\mcp-cli.exe` (Bun, mcp-cli v0.3.0)
- Конфиг: `~\.mcp_servers.json` (сервер `zai-vision`, модель GLM-4.6V)
- Работает из любой директории

**Перед каждым вызовом обновить PATH:**
```powershell
$env:Path = [System.Environment]::GetEnvironmentVariable("Path","User") + ";" + [System.Environment]::GetEnvironmentVariable("Path","Machine")
```

**Команды:**
```powershell
# Анализ изображения
mcp-cli call zai-vision analyze_image '{"image_source":"C:/path/file.png","prompt":"Опиши что видишь"}'

# OCR — распознавание текста
mcp-cli call zai-vision extract_text_from_screenshot '{"image_source":"C:/path/file.png","prompt":"Извлеки текст"}'

# Сравнение двух скриншотов
mcp-cli call zai-vision ui_diff_check '{"expected_image_source":"ref.png","actual_image_source":"cur.png","prompt":"Сравни"}'

# Диагностика ошибки
mcp-cli call zai-vision diagnose_error_screenshot '{"image_source":"err.png","prompt":"Что за ошибка"}'

# Список серверов и инструментов
mcp-cli -d
```

**Правила:**
- JSON-аргументы в одинарных кавычках: `'{"key":"value"}'`
- Пути с `/` (forward slash), не `\`
- Каждый вызов ≈ 10–15 секунд
- Сначала пробовать встроенные инструменты, потом mcp-cli как fallback

**Обёртка с таймаутом:** `scripts/dev/vision.ps1 -Tool <name> -Json '<json>' -TimeoutSec 90` — убивает mcp-cli при зависании (см. §12.8).

**Как использовать в рабочем цикле:**
1. Сделать скриншот: `gameActions.snapshot('bug_report')` или `godot.takeScreenshot()`
2. Скриншоты сохраняются в `tests/e2e/screenshots/`
3. Передать скриншот Vision MCP (способ 1 или 2)
4. MCP проанализирует и вернёт описание/диагноз
5. Агент чинит на основе визуального анализа

### 12.8 Правило таймаутов и авто-восстановления

**ЛЮБАЯ** команда (build / serve / test / vision / godot), которая может зависнуть, **ОБЯЗАНА** запускаться с таймаутом. Если процесс не уложился — убивать и сообщать где зависло, чтобы агент мог восстановиться в автоматическом режиме.

**Dev-скрипты** (в `scripts/dev/`):

| Скрипт | Назначение | Таймаут по умолчанию |
|--------|------------|----------------------|
| `build.ps1` | HTML5 export через Godot headless. Kill если зависнет | 300с |
| `serve.ps1` | http-server на :3000 с авто-рестартом при падении (≤20 рестартов) | watchdog 5с poll |
| `test.ps1` | jest e2e с per-test timeout, авто-rerun flake, общий timeout | 1500с overall, 120с/тест |
| `vision.ps1` | Обёртка над `mcp-cli call zai-vision ...` | 90с |
| `verify.ps1` | Сборка→сервер→smoke→vision главной страницы | 300с |

**Примеры:**
```powershell
# Сборка с таймаутом
& scripts/dev/build.ps1 -TimeoutSec 300

# Тесты с авто-flake-rerun
& scripts/dev/test.ps1 -Suite "user/" -OverallTimeoutSec 1500

# Vision анализ скриншота
& scripts/dev/vision.ps1 -Tool analyze_image -Json '{"image_source":"C:/path/s.png","prompt":"Опиши"}' -TimeoutSec 90
```

**Принципы авто-восстановления:**
1. Каждый шаг логирует начало/конец/длительность
2. При timeout — kill + exit code 124 + продолжить следующий шаг
3. При краше сервера — авто-рестарт (≤20 раз), потом fail
4. При flake теста — один rerun, потом accept fail
5. Никаких `Start-Process ... -Wait` без `WaitForExit($ms)` — только параметризованные таймауты
6. Все логи в `$env:TEMP\bookwar_*.log` — агент может их прочитать

## 13. ТЕСТИРОВАНИЕ

### 13.1 Стратегия тестирования

Тестирование проводится **исключительно через реальную игру** (HTML5 build + Puppeteer), как это делает живой игрок. Компонентные, пользовательские и regression-тесты — **все** работают через один и тот же путь: запускают Godot HTML5 в headless Chromium, двигают игрока клавиатурой/мышью, читают `window.*` переменные моста.

**КЛЮЧЕВЫЕ ПРАВИЛА (НАРУШЕНИЕ = БАГ):**
1. **НИКОГДА не вызывай внутренние GDScript-функции из тестов.** Нет `combat_system.calculate_damage()`, нет `new Inventory()`, нет прямого импорта GDScript-классов. Единственный интерфейс — Puppeteer + `window.*`.
2. **Test-bridge** (`window.gameStartTestCombat`, `window.gameTestAddLetter` и т.п.) — единственное исключение, и только для **детерминированного сетапа** (задать врага, выдать букву). Само **действие игрока** (атака, движение, подбор) проходит через тот же ввод, что и реальная игра.
3. **Каждое изменение визуала/геймплея → скриншот + Vision-MCP анализ** (см. §12.4 GATE 2). Никогда не утверждай «должно работать» — сделай скрин, передай в `mcp-cli call zai-vision analyze_image`, прочитай ответ.
4. **Бот-прохождение** (`walkthrough.test.js`) — эталонный сценарий «как играть». Каждый шаг: действие → ожидаемый результат → таймаут → если зависло, дамп состояния + диагноз причины. Этот шаблон повторяется для каждого региона.
5. **Зачистка процессов**: перед запуском нового сервера ВСЕГДА убивай старый (`Get-NetTCPConnection :3000 | Stop-Process`). После теста ВСЕГДА `Stop-Job` + `Remove-Job` + kill порта. Не оставляй висящие процессы — они копятся и блокируют порт.
6. **Таймауты на ВСЁ**: build (300с), serve (30с подъём), test (90с/тест), vision (90с). Команда без таймаута = потенциальный вечный вис.

### 13.11 Бот-прохождение (Walkthrough Framework)

`walkthrough.test.js` — эталон «как пройти регион». Каждый регион (Светлая Долина, Тёмный Лес, ...) получает свой walkthrough по этому шаблону:

1. **Каждый шаг** описан: действие игрока → ожидаемый результат → таймаут (мс)
2. **Логирование**: каждый шаг печатает `Шаг NN — действие → ✓ OK (мс)` или `✗ ЗАВИСЛО`
3. **Детектор зависания**: если ожидаемый результат не наступил за таймаут → дамп `window.*` состояния + скриншот `STUCK_шагNN.png` + диагноз причины (`diagnose()`)
4. **Скриншот** после каждого шага для Vision-анализа

**Шаблон шага:**
```javascript
await runStep('Описание что делает игрок',
  async () => { /* действие: movePlayer, interact, selectBattleCard... */ },
  (st) => ({ ok: /* проверка st.* */, detail: 'человекочитаемый результат' }),
  15000  // таймаут мс

```

**Правило**: при зависании бот НЕ падает молча — он логирует полное состояние игры и причину, чтобы агент сразу видел что чинить.

### 13.3 Принципы пользовательских тестов (User Tests)

Пользовательские тесты моделируют действия реального игрока:
- **Не вызывать внутренние функции** напрямую (нет `combat_system.calculate_damage()`)
- **Действовать через UI/ввод**: кликнуть кнопку, нажать клавишу, переместить мышь
- **Проверять видимый результат**: что на экране отображается ожидаемый результат
- Каждый тест описывает сценарий с точки зрения игрока

Пример:
```javascript
// ПРАВИЛЬНО — пользовательский тест
test('игрок подбирает точку и видит её в инвентаре', async () => {
  await gameActions.movePlayerTo(page, dotPosition);
  await gameActions.interact();
  const inventory = await gameActions.getInventoryContents();
  expect(inventory.dots).toBeGreaterThan(0);
});

// НЕПРАВИЛЬНО — вызов внутренней функции
test('inventory adds dot', () => {
  const inv = new Inventory();
  inv.add_item('dot');
  expect(inv.has_item('dot')).toBe(true);
});
```

### 13.4 Принципы компонентных тестов (Component Tests)

Компонентные тесты проверяют изолированные подсистемы **через реальную игру в браузере** (не через внутренние вызовы!):
- Боевая система: инициировать бой через `window.gameStartTestCombat` → выбрать карту через `gameSelectCard` → подтвердить через `gameConfirmTurn` → прочитать результат в `window.gameCombatLogAll`
- Инвентарь: открыть через клавишу I → прочитать `window.gameInventory`
- Монстры: подойти к монстру → прочитать `window.gameMonsterStates`
- HUD: прочитать `window.gameHUD`
- Алфавит/формулы: прочитать `window.gameAlphabet` (33 буквы, `base_power = 34 - position`, типы)

**Component-тесты НЕ импортируют GDScript.** Они работают в том же Puppeteer-окружении, что и user-тесты. Разница только в фокусе: component проверяет одну подсистему, user — сценарий игрока целиком.

## 15. ДОПОЛНИТЕЛЬНЫЕ ЗАМЕТКИ

### 15.1 Баланс (предварительный)

- Медленные буквы (А, Б) — мощные, но ходят последними. Риск: получишь урон до хода.
- Быстрые буквы (Э, Ю, Я) — слабые, но ходят первыми. Хорошо для добивания.
- Оптимальная стратегия: баланс скорости и силы в колоде.
- Ъ + согласная = усиленная защита (комбо из 2 карт)
- Ь + гласная = усиленная атака (комбо из 2 карт)

### 15.2 Идеи на будущее

- **Английский алфавит**: Второй набор букв для расширения
- **Рейтинг игроков**: На основе собранного алфавита
- **Ивенты**: Временные события с редкими буквами
- **Крафт**: Комбинация букв для создания новых знаков препинания

### 15.3 Генерация ассетов через Pollinations.ai

> **ПРАВИЛО**: красивые графические ассеты (глифы букв, заставки, иконки) генерируются
> через **Pollinations.ai** — бесплатный API генерации изображений по текстовому промпту.
> НЕ оставлять буквы как plain-текст — генерировать декоративные глифы в стиле
> старославянских рукописей / рунических карточек.

**API** (возвращает PNG напрямую):
```
https://image.pollinations.ai/prompt/{URL-encoded-prompt}?width=256&height=256&nologo=true&seed=42
```

**Промпт-шаблон для букв** (каждая буква — отдельный PNG):
```
ornate medieval Slavic manuscript letter {БУКВА}, golden calligraphy on aged parchment,
dark fantasy card game glyph, intricate border, candle light, highly detailed, centered
```

**Скачивание (PowerShell):**
```powershell
$prompt = [System.Web.HttpUtility]::UrlEncode("ornate medieval Slavic manuscript letter А ...")
Invoke-WebRequest -Uri "https://image.pollinations.ai/prompt/$prompt?width=256&height=256&nologo=true&seed=1" -OutFile "assets/sprites/letters/glyph_А.png"
```

**Что генерировать:**
- 33 глифа букв (А...Я) — `assets/sprites/letters/glyph_{БУКВА}.png`
- Глифы Ъ и Ь — с особым свечением (синее/красное)
- Заставки регионов — `assets/sprites/splash/{region}.png`
- Иконки заклинаний — `assets/sprites/spells/{word}.png`

**Интеграция в карточки**: `letter_card.tscn` использует `TextureRect` с `glyph_{БУКВА}.png`
вместо plain `Label` — буквы выглядят как расписные старославянские письмена.

## 16. ЗАКЛИНАНИЯ — БОЕВЫЕ КОМБИНАЦИИ БУКВ

> Ключевая механика углубления боя. Буквы — это не только оружие/защита по отдельности,
> но и **компоненты заклинаний**. Собрав нужные буквы, игрок складывает из них **слова** —
> мощные комбинированные эффекты, сильнее чем отдельные буквы.

### 16.1 Принцип

- **Заклинание = короткое слово** из 2-4 букв (БАМ, ЩИТ, БОГ, ЖАР, ...)
- Игрок **открывает (покупает)** рецепт заклинания за валюту (см. §16.4)
- Открыв заклинание, игрок может **скастовать** его в бою, если имеет все нужные буквы (уровень ≥ 1)
- Каждое заклинание **стоит 1 ход** (как одна буква), но эффект комбинированный и усиленный
- Буквы при касте **НЕ расходуются** (уровень постоянный) — заклинание использует их силу

### 16.2 Стоимость и валюта

**Буквы = валюта.** Каждая буква имеет ценность = позиция в алфавите:
- А = 1 (самая дешёвая), Я = 33 (самая дорогая)
- **Кошелёк игрока** = Σ (позиция × уровень) по всем собранным буквам
- Пример: А(ур.3) + О(ур.2) + Я(ур.1) = 1×3 + 16×2 + 33×1 = 3+32+33 = **68 монет**

**Стоимость открытия заклинания** = Σ позиций букв × множитель_рецепта (обычно ×3):
- БАМ = (2+1+14)×3 = 17×3 = **51 монета**
- ЩИТ = (27+10+20)×3 = 57×3 = **171 монета** (дороже — мощнее)

### 16.3 Реестр заклинаний (data/spells.json)

| Слово | Буквы | Эффект | Тип | Множитель |
|-------|-------|--------|-----|-----------|
| **БАМ** | Б+А+М | Усиленная атака (урон ×2.0) | атака | 2.0 |
| **БУМ** | Б+У+М | Взрыв — игнорирует щит врага | атака_пробой | 1.8 |
| **ЩИТ** | Щ+И+Т | Двойной щит (защита ×2.5) | защита | 2.5 |
| **БОГ** | Б+О+Г | Лечение +40 HP | лечение | 1.0 |
| **ДЫМ** | Д+Ы+М | Дымовая завеса — уклон от след. атаки | уклон | 1.0 |
| **ЖАР** | Ж+А+Р | Поджог — урон 3 хода подряд | DoT | 1.5 |
| **ЛУК** | Л+У+К | Дальний выстрел — враг не отвечает | атака_без_ответа | 1.6 |
| **РОГ** | Р+О+Г | Призыв — бафф ×2 на след. ход | бафф | 2.0 |
| **ХРЕН** | Х+Р+Е+Н | Мощь — тройной удар по врагу | атака_массовая | 2.2 |
| **ФИГ** | Ф+И+Г | Контр-удар — отражает след. атаку | контр | 1.0 |

**Формула силы заклинания:**
```
spell_power = (Σ base_power букв) × spell_multiplier × средний_уровень_букв
```
Пример БАМ (Б=32, А=33, М=20, все ур.1): (32+33+20) × 2.0 × 1 = 170 урона — намного сильнее чем А=33 отдельно.

### 16.4 Механика в бою

1. В сцене боя, помимо кнопок отдельных букв, появляется панель **заклинаний** (только открытые)
2. Игрок выбирает: отдельная буква ИЛИ заклинание (одно за ход)
3. При выборе заклинания — проверка наличия всех букв → эффект применяется как единое действие
4. Заклинание ходит по **скорости самой медленной буквы** в слове (баланс: мощные слова медленные)
5. Покупка заклинаний — в **деревне/цитадели** у NPC «Кузнец Слов» (см. §6.2)

## 17. ЛОЯЛЬНОСТЬ МОНСТРОВ, БАНДА, ПОБЕДА, СКРЫТИЕ

> Новые механики (добавлены 2026-06-15). Реализованы в `monster_base.gd`, `world_map.gd`,
> `monster_spawner.gd`, `constants.gd`, `game_state.gd`, `hud_ui.gd`, `player.gd`.

### 17.1 Лояльность монстров (Allegiance)

Каждый монстр имеет одно из трёх состояний лояльности:

| Состояние | Цвет | Поведение |
|-----------|------|----------|
| **HOSTILE** (0) | 🔴 Красный | Преследует и атакует игрока |
| **RECRUITED** (1) | 🟢 Зелёный | Следует за героем, сражается за него |
| **NEUTRAL** (2) | ⚪ Серый | Игнорирует игрока, патрулирует |

- Стартовое состояние: `aggressive` монстры (`!`) → HOSTILE. `patrol` монстры (`?`) → NEUTRAL.
- Цвет применяется через `modulate` на спрайте/лейбле монстра.
- Размер монстра = размер героя (`MONSTER_SCALE = 1.6`, `MONSTER_LABEL_SIZE = 48`).

### 17.2 Рекрутинг (50/50)

После **диалога** с монстром (если есть многоточие `...`):
1. Игрок жмёт **T** рядом с монстром → начинается диалог.
2. Расходуется 1 многоточие.
3. Игрок жмёт **E** или **T** → диалог закрывается → **бросок 50/50**:
   - **Успех (50%)**: монстр становится **RECRUITED** (зелёный). Говорит: «Возможно ты и не враг... Я с тобой!» Бегает за героем.
   - **Провал (50%)**: 
     - `!` монстр остаётся **HOSTILE** (красный). Говорит: «Нет! Ты враг!» Возвращается к преследованию.
     - `?` монстр становится **NEUTRAL** (серый). Говорит: «Ладно, иди. Я нейтрален.»

### 17.3 Банда (Gang)

- Завербованные монстры **следуют за героем** с небольшим случайным оффсетом.
- Состояние: `"follow"`. Логика: `_process_follow()` в `monster_base.gd`.
- Чем больше банда — тем престижнее игрок.
- **В будущих версиях**: банда участвует в бою на стороне игрока.

### 17.4 Условия победы в регионе (Victory)

**Светлая Долина считается пройденной**, когда выполнено одно из условий для **КАЖДОГО** монстра:
- **Убит** в бою
- **Завербован** (присоединился к банде)
- **Нейтрален** (переболтал)

**Обязательное условие**: игрок должен **подойти** к каждому монстру хотя бы один раз (флаг `_approached`).

Проверка выполняется периодически в `world_map.gd._check_victory()`.
При победе: на экране «ПОБЕДА! Долина зачищена. Союзников: N».

### 17.5 Скрытие в тёмных тайлах (Hiding)

Игрок может **спрятаться** от преследующих монстров, встав на:
- 🌲 **Лес** (TREE, source 1)
- 💧 **Вода** (WATER, source 2)
- 🪨 **Камень** (HOUSE/STONE, source 3)
- 🚧 **Забор** (FENCE/STONE, source 5)

На траве (GRASS, 0) и тропе (PATH, 4) — **открыто**, спрятаться нельзя.

Проверка: `world_map.gd._check_player_hidden()` — проверяет тайл под игроком, ставит `GameState.player_hidden`.

### 17.6 Поиск и потеря интереса (Search → Give Up)

Когда монстр преследует (`chase`) и:
1. **Игрок спрятался** в тёмном тайле → монстр переходит в `search`.
2. **Игрок вышел из зоны обнаружения** → монстр переходит в `search`.

В состоянии `search` монстр:
- Стоит, оглядывается `SEARCH_DURATION` секунд (4с).
- Если игрок снова попал в радиус обнаружения и не скрыт → возобновляет погоню.
- Если время вышло → **теряет интерес**, возвращается к патрулированию.

### 17.7 Диалог — авто-закрытие

**Диалог автоматически закрывается**, если:
1. Игрок нажал **E** или **T** (закрыть диалог).
2. **Игрок отошёл** от монстра (вышел из зоны обнаружения) — `_on_detection_body_exited` вызывает `GameState.end_dialogue()`.

При закрытии диалога срабатывает рекрутинг 50/50 (§17.2).

### 17.8 Пауза (Pause)

- **Пробел (Space)** — переключает паузу.
- При паузе: `get_tree().paused = true`, на экране «⏸ ПАУЗА».
- HUD имеет `process_mode = PROCESS_MODE_ALWAYS` — обрабатывает ввод даже в паузе.

## 18. ПРОГРЕССИЯ МИРОВ, КВЕСТЫ, ТОРГОВЛЯ, ЭВОЛЮЦИЯ ВРАГОВ

> Ключевой дизайн-документ. Описывает как игрок продвигается от карты к карте,
> какие квесты и механики доступны на каждом уровне, как враги эволюционируют.
> Добавлено 2026-06-15.

### 18.1 Цепочка карт (Map Progression Chain)

Игра — последовательность **5 карт-миров**, каждая сложнее предыдущей.
Между картами — **зелёный коридор** (transition zone), открывается когда **все квесты** текущей карты выполнены.

```
[1] Светлая Долина  →  [2] Лесная Поляна  →  [3] Дремучие Дубы  →  [4] Ивовая Гать  →  [5] Болотная Топь
   (старт, обучающая)     (первые NPC)         (опасный лес)         (болото, туман)     (финал, босс)
```

**Коридор (переход)**:
- Визуально: светящаяся зелёная дорожка на северном краю карты
- Появляется **только** когда все квесты карты выполнены
- Игрок входит в коридор → загружается следующая карта
- Назад возвращаться **можно** (коридор остаётся открытым)

### 18.2 Параметры прогрессии по картам

| # | Карта | Биом | Врагов | Квестов | Букв доступно | Особенность |
|---|-------|------|--------|---------|---------------|-------------|
| 1 | **Светлая Долина** | Луга, солнце | 8-10 (? и !) | 3-4 | А, О, М, Б, Я (первые 5) | Обучение: сбор точек, диалог, первый бой |
| 2 | **Лесная Поляна** | Светлый лес | 12-15 (! + NPC) | 5-6 | + Е, К, Т, Р, Д | Первый **реальный NPC** (Длинноязыкий), торговля |
| 3 | **Дремучие Дубы** | Тёмный лес | 18-20 | 7-8 | + В, Г, Ж, Л, П, Н | Враги с **большими ушами** (Слушач), больше HP |
| 4 | **Ивовая Гать** | Болото, мосты | 20-25 | 8-10 | + З, С, Ф, Х, Ц, Ч, Ш, Щ | Враги с **большими глазами** (Зрячий), щиты, заклинания |
| 5 | **Болотная Топь** | Глубокое болото | 25-30 + босс | 10-12 | + Й, Ъ, Ы, Ь, Э, Ю, Ё (все 33) | Враги с **большим ртом** (Жор), **босс — Хранитель Топи** |

**Принцип**: каждая следующая карта добавляет ~30% врагов, квестов и букв.
К концу 5-й карты игрок собирает **все 33 буквы**.

### 18.3 Квестовая система (`?` монстры = квестодавцы)

**Только `?` монстры дают квесты.** `!` монстры — только бой/преследование.

#### Типы квестов:

| Тип | Описание | Пример |
|-----|----------|--------|
| **collect** | Принести N букв определённого типа | «Принеси 3 буквы А» → награда: буква О |
| **defeat** | Убить N врагов `!` | «Очисти поляну от 3 врагов» → награда: заклинание |
| **talk** | Поговорить с другим `?` монстром | «Передай привет Мудрецу» → награда: точки |
| **trade** | Обменять одну букву на другую | «Дам тебе К за твою Я» → мгновенный обмен |
| **buy** | Купить букву за точки | «Купи букву М за 10 точек» |

#### Структура квеста в `data/monsters.json`:

```json
{
  "id": "quest_giver_1",
  "name": "Сомнение",
  "behavior": "patrol",
  "quests": [
    {
      "id": "q_collect_a",
      "type": "collect",
      "description": "Принеси мне 3 буквы А. Я дам тебе О.",
      "requirement": {"letter": "А", "count": 3},
      "reward": {"type": "letter", "letter": "О", "count": 1}
    },
    {
      "id": "q_buy_m",
      "type": "buy",
      "description": "Хочешь букву М? 10 точек.",
      "cost": {"resource": "dots", "amount": 10},
      "reward": {"type": "letter", "letter": "М", "count": 1}
    },
    {
      "id": "q_trade_ya_k",
      "type": "trade",
      "description": "Меняю мою К на твою Я.",
      "give": {"letter": "К"},
      "receive": {"letter": "Я"},
      "reward": {"type": "letter", "letter": "К", "count": 1}
    }
  ]
}
```

#### Механика диалога с квестами:

1. Игрок подходит к `?` монстру → жмёт **T**
2. Открывается диалог с **меню квестов** (а не только текст)
3. Игрок выбирает квест → видит условие и награду
4. **Принять** → квест добавляется в журнал
5. Выполнить условие → вернуться к квестодавцу → **сдать** → получить награду
6. Для **buy/trade** — мгновенное выполнение (если хватает ресурсов)

#### Условие открытия коридора:

```gdscript
func _check_all_quests_done() -> bool:
    for quest in all_quests_on_map:
        if not quest.is_completed():
            return false
    return true
```

Когда все квесты выполнены → спавнится **PortalNode** на северном краю.

### 18.4 Торговля (Trade & Shop)

**`?` монстры могут быть торговцами** (shop type):

| Действие | Описание | Условие |
|----------|----------|---------|
| **Купить букву** | Отдать N точек → получить букву | Хватает точек |
| **Обменять букву** | Отдать букву X → получить букву Y | Есть буква X, уровень ≥ 1 |
| **Купить заклинание** | Отдать N точек → выучить рецепт | Хватает точек |

**Баланс цен** (точки = единственная валюта на старте):

| Покупка | Цена в точках |
|---------|---------------|
| Редкая буква (А, Б, В) | 15-20 |
| Обычная буква (М, К, Т) | 8-12 |
| Частая буква (Я, Ю, Э) | 3-5 |
| Заклинание (рецепт) | 30-50 |

**Обмен**: курс зависит от разницы позиций в алфавите.
- А(1) ↔ Я(33) = невыгодный курс (33:1), нужен доплата точками
- Б(2) ↔ В(3) = честный курс (1:1)

### 18.5 Эволюция врагов — от знаков к существам

**Принцип**: с каждой новой картой враги становятся **визуально разнообразнее**.
Вместо абстрактных `?` и `!` — **рисованные персонажи** с уникальной внешностью.

#### Карта 1: Светлая Долина — только знаки
- `?` (вопрос) — патрулирующие нейтралы, квестодавцы
- `!` (восклицание) — агрессивные, атакуют

#### Карта 2: Лесная Поляна — первый реальный NPC
**Длинноязыкий** (Longtongue):
- Тело как у игрока (гуманоид), но зелёного оттенка
- **Длинный язык** свисает изо рта (визуальная фишка)
- Поведение: хитрый торговец, даёт квесты на обмен букв
- HP: средний, в бою атакует языком (тип: vowel)
- Рисуется через **Polygon2D** (не Label): тело + голова + язык

#### Карта 3: Дремучие Дубы
**Слушач** (Big-Ears):
- Гуманоид с **огромными ушами** (в 2 раза больше головы)
- Слышит игрока далеко (detection_radius × 2)
- Поведение: агрессивный, быстрый
- HP: высокий, в бою защищается (тип: consonant)

#### Карта 4: Ивовая Гать
**Зрячий** (Big-Eyes):
- Существо с **огромными глазами** (занимают половину лица)
- Видит игрока сквозь укрытия (игнорирует hiding)
- Поведение: хитрый, атакует заклинаниями
- HP: средний, использует баффы (Ъ/Ь)

#### Карта 5: Болотная Топь
**Жор** (Big-Mouth):
- Огромный **рот** занимает всё тело
- Сильная атака, поглощает щиты
- Поведение: танк, медленный но мощный
- HP: очень высокий

**Босс: Хранитель Топи**:
- Огромная версия Жора
- Владеет всеми 33 буквами
- Несколько фаз боя
- При победе: финальная концовка

#### Принцип отрисовки NPC (для AI-агента):

Каждый NPC рисуется **программно через Polygon2D** (без внешних спрайтов):
```gdscript
func _draw_npc(npc_type: String) -> void:
    # Тело (общее для всех)
    var body := Polygon2D.new()
    body.polygon = [Vector2(-15,-30), Vector2(15,-30), Vector2(15,15), Vector2(-15,15)]

        "big_eyes":   _draw_big_eyes()  # гигантские глаза
        "big_mouth":  _draw_big_mouth() # огромный рот
```

## 19. ВОБЛ (WADDLE) И ТРЕВОЖНЫЙ ВОБЛ (ALERT WOBBLE)

> Анимация «переваливания» для всех движущихся объектов + индикатор опасности/сокровищ.
> Добавлено 2026-06-22. Реализовано в `player.gd` и `monster_base.gd`.

### 19.1 Принцип вобла (обычное переваливание)

**КАЖДЫЙ** объект, который перемещается по карте, слегка «переваливается» при движении:
- Лёгкий наклон влево-вправо (rotation) + маленькое вертикальное подбрасывание (bob).
- Не быстро и не мгновенно — мягкое, плавное покачивание.
- Применяется ко всем: герой, `?`, `!`, нарисованные твари (forest_creature, dark_wolf, shadow_lurker), **шестёрки колдуна** (Знак, Звук), завербованные союзники.

**Реализация (единая для всех):**
- Визуальные части тела (Polygon2D + Label) складываются под **`_visual_root: Node2D`** — pivot, сидящий в начале координат сущности.
- Вобл применяется **ТОЛЬКО к `_visual_root`**. Физика (collision), зоны (interaction/detection) и **камера** остаются стабильными — ничего не трясётся, кроме картинки.
- В `_physics_process` считается `_walk_phase += delta * WOBBLE_FREQ`, и:
  ```
  _visual_root.rotation = sin(phase) * WOBBLE_AMP
  _visual_root.position.y = -abs(sin(phase)) * WOBBLE_BOB
  ```
- При остановке — плавный `lerp` обратно к нулю (`delta * 10.0`).

**Константы:**
| Сущность | WOBBLE_AMP | WOBBLE_FREQ | WOBBLE_BOB |
|----------|-----------|-------------|------------|
| Герой (`player.gd`) | 0.10 рад (~5.7°) | 11.0 | 1.6 px |
| Монстры (`monster_base.gd`) | 0.11 рад (~6.3°) | 9.0 | 1.5 px |

### 19.2 Тревожный вобл (Alert Wobble) — индикатор опасности/сокровищ

Вобл **удваивается** (×2 и амплитуда, и частота) в двух случаях:

#### 19.2.1 Монстр заметил героя
Когда монстр переходит в состояния **`chase` / `suspicion` / `search`** (заметил, ищет, преследует):
- Он **«встрепенулся»** — вобл становится 2× сильнее/быстрее.
- Вобл идёт **на месте**, даже если `velocity == 0` (т.е. во время `suspicion`/`search`, когда монстр стоит и оглядывается).
- Это **визуальный индикатор для игрока**: «опасность, этот враг тебя засёк».
- Реализация: `_update_wobble(delta, moving, alert=true)` — флаг `alert` форсирует осцилляцию и множитель `WOBBLE_ALERT_MULT = 2.0`.

#### 19.2.2 Герой чувствует опасность или сокровище
Герой тоже переходит на 2× вобл, когда:
- **Враг виден на краю экрана** (в радиусе `ALERT_SCREEN_DIST = 360px`, но **дальше** `ALERT_NEAR_DIST = 130px`) — «чуешь угрозу неподалёку».
- **Залежи буквиц**: в радиусе `TREASURE_RADIUS = 200px` собралось **≥ `TREASURE_THRESHOLD = 7`** буквиц-валюты — «чуешь сокровище».

**Исключение:** если враг **вплотную** (`≤ ALERT_NEAR_DIST`) — тревожный вобл **отключается**, остаётся обычный (опасность уже очевидна, дёргаться незачем).

**Скан** соседних врагов/буквиц идёт с throttling `ALERT_SCAN_INTERVAL = 0.15с` (не каждый кадр) — производительность.

### 19.3 Правила для AI-агента

- **НЕ применять вобл к камере** — только к `_visual_root`. Иначе весь экран наклоняется → тошнота.
- **НЕ вращать сам CharacterBody2D** героя (у него дочерняя Camera2D) — только визуальный pivot.
- При добавлении НОВОГО движущегося персонажа — обязательно добавить `_visual_root` + `_update_wobble`.
- Шестёрки колдуна: **на карте 2 — ровно 1** (Знак), **на карте 3 — 2** (Знак + Звук). См. `monster_spawner.gd`.
- Любое изменение визуала/движения → скриншот + Vision-MCP (§12.4 GATE 2), чтобы убедиться что вобл виден и не ломает камеру/UI.

### 20.1 Новая формула силы букв (ИНВЕРСИЯ §2.1)

**Старое правило (устарело)**: `base_power = 34 − position` (А=33 сила, Я=1 сила).
**Новое правило (согласно §20)**:

```
letter_coefficient = position_in_alphabet
  А = 1   (самая слабая)
  ...
  Я = 33  (самая сильная)
```

**Сила = позиция в алфавите.** Буквы ближе к концу алфавита (Ю, Э, Я, Ъ, Ы, Ь) — мощные.
Буквы ближе к началу (А, Б, В, Г, Д) — слабые.

Это **полная инверсия** прежнего правила. Требует обновления:
- `data/letters.json` — у всех 33 букв `base_power = position` (НЕ `34 − position`)
- `scripts/core/alphabet_data.gd` — формула `get_base_power()`
- Все формулы боя в `combat_system.gd` (см. §2.3) — переписать через новый коэффициент
- Спеллы в `data/spells.json` — формула силы тоже опирается на новый коэффициент
- Описание букв в `data/letters.json` — обновить тексты («мощная/слабая»)

### 20.2 Редкость букв по уровням

Распределение дропа/спавна букв подчиняется правилу:

| Уровень (карта) | Максимум копий одной буквы на карте |
|-----------------|--------------------------------------|
| 1 (Светлая Долина) | 2 |
| 2 (Лес Двубуквия) | 3 |
| 3 (Дремучие Дубы) | 3 |
| N (где N ≥ 4) | N |

Плюс правило редкости:
- **Частые** (позиции 1-10, А-И): падают с весом ×3
- **Обычные** (позиции 11-22, Й-У): падают с весом ×2
- **Редкие** (позиции 23-33, Ф-Я): падают с весом ×1

Реализация: в `world_map.gd._spawn_*_items` и в `monster_base._drop_loot` использовать
взвешенный выбор буквы вместо равновероятного.

### 20.3 Новый бой: тело-экипировка (тактический режим)

Параллельно со старым автобусоем — **схема тела героя** с слотами под буквы:

```
        [Шлем]           ← согласная (голова)
         ⬛
   ⬛⬛⬛[Корпус]⬛⬛⬛    ← согласная (кольчуга/торс)
   ⬛   ⬛⬛⬛   ⬛
  [Л.рука]       [П.рука]
   ⬛              ⬛       ← П.рука = гласная (оружие)
   ⬛              ⬛       ← Л.рука = согласная (щит) ИЛИ ещё одна гласная
```

**Слоты и формулы:**
- **Правая рука** — гласная (оружие). Может быть две гласные (два оружия).
- **Левая рука** — согласная (щит) ИЛИ гласная (второе оружие).
- **Корпус** — согласная (кольчуга).
- **Голова** — согласная (шлем).
- **Очки атаки** = Σ коэффициентов гласных в руках.
- **Очки брони** = Σ коэффициентов согласных в щите/корпусе/шлеме.

**Тумблер режимов** (кнопка в `battle_manager.gd`, уже есть «Автобой»):
- **Автобой ВКЛ** — текущий card-based бой (как сейчас).
- **Автобой ВЫКЛ** — тактический режим с телом-экипировкой.

В автобое используется та же фигурка героя, но исход боя resolves автоматически
из текущих очков атаки/брони экипировки.

### 21.5 ИЗВЕСТНЫЕ БАГИ GODOT 4.6.3 + WORKAROUND'ы

> Эти баги заняли часы отладки. Записано чтобы не повторять.

1. **`can_export()` возвращает false без объяснения** в CLI. UI показывает preset валидным,
   CLI падает с "configuration errors: " (пусто после двоеточия).
   **Workaround**: включить в `project.godot`:
   ```
   [rendering]
   textures/vram_compression/import_etc2_astc=true
   ```

2. **`.build_version` файл обязателен** в `android/.build_version` (на уровне ВЫШЕ build/!).
   Содержимое: `4.6.3.stable`. Без него export падает с
   "Trying to build from a gradle built template, but no version info for it exists."

3. **Gradle wrapper download timeout** — `gradle-wrapper.properties` по умолчанию тянет
   `gradle-8.11.1-bin.zip` с `services.gradle.org`, иногда timeout.
   **Workaround**: использовать локальный gradle от Android Studio:
   ```powershell
   $src = "D:\AndroidStudioData\gradle\wrapper\dists\gradle-8.12-all\ejduaidbjup3bmmkhw3rie4zb\gradle-8.12-all.zip"
   $dst = "D:\Projects\BOOKWAR\android\build\gradle\wrapper\gradle-8.12-all.zip"
   Copy-Item $src $dst -Force
   "distributionUrl=file:///$($dst -replace '\\','/')" | Out-File gradle-wrapper.properties -Encoding ASCII
   ```
   `build_apk.ps1` делает это автоматически.

4. **Install Android Build Template** НЕ создаётся при CLI export автоматически.
   Только через UI меню Project → Install Android Build Template.
   **Workaround**: используй `install_template_mouse.py` (pywinauto):
   ```powershell
   python "D:\Projects\BOOKWAR\ANDROID_VERSION\scripts\dev\install_template_mouse.py"
   ```

5. **Vulkan на эмуляторе падает** с `GL_MAX_FRAGMENT_UNIFORM_VECTORS exceeded` (SwiftShader).
   **Workaround**: в `project.godot`:
   ```
   [rendering]
   renderer/rendering_method="gl_compatibility"
   renderer/rendering_method.mobile="gl_compatibility"
   ```
   И запускать эмулятор с `-gpu host` (через `run_emulator.ps1`).

6. **Stretch aspect="keep" ломает touch coordinates** — на эмуляторе с разным DPI/aspect
   координаты transforms неправильно.
   **Workaround**: использовать `aspect="expand"` в `project.godot` (или `ignore`).
