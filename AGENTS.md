## 1. ОБЗОР ПРОЕКТА

**Название**: BOOKWAR
**Жанр**: RPG с карточными боями на основе русского алфавита, мрачное средневековье
**Движок**: Godot 4.x (GDScript)
**Платформы**: Web (itch.io), PC (Windows/Mac/Linux), Mobile (Android/iOS)
**Приоритет платформ**: itch.io (HTML5 export) — главное требование, должно работать в браузере
**Режимы**: Одиночная игра + мультиплеер (кооп и PvP)

**Кратко**: В мире запретили буквы. Разбросали по тёмным землям. Игрок — герой, который отправляется вернуть все 33 буквы русского алфавита. Буквы — это одновременно оружие (гласные = атака), защита (согласные = щит), модификаторы (Ъ, Ь) и валюта. Знаки препинания — особые способности. Мир — единая огромная карта мрачного средневековья, начинающаяся с уютной светлой долины и уходящая во тьму.

## 2. БУКВЕННАЯ СИСТЕМА — ЯДРО ИГРЫ

**Принцип баланса**: Медленные буквы (начало алфавита) — мощные, но ходят последними. Быстрые буквы (конец алфавита) — слабые, но ходят первыми.

### 2.2 Уровень буквы = количество копий

У игрока нет «уровня» в привычном смысле. У каждой буквы есть свой уровень.

- **Уровень буквы = количество собранных копий этой буквы**
- 1 копия А = уровень 1. Урон = 33 × 1 = 33
- 3 копии А = уровень 3. Урон = 33 × 3 = 99
- 5 копий Щ = уровень 5. Защита = 7 × 5 = 35

**Продажа/обмен**: Игрок может отдать 1 копию буквы другому игроку или NPC. При этом его уровень этой буквы уменьшается на 1, а у получателя — увеличивается на 1.

**Использование в бою**: При атаке/защите буква НЕ теряется. Уровень буквы — это перманентная характеристика. Но если продал копию — сила падает.

### 2.3 Формулы боя

```
base_power = 34 - speed_position    (А=33, Я=1)
speed = speed_position              (А=1, Я=33)

Для ГЛАСНЫХ (атака):
  damage = base_power × letter_level × VOWEL_MULTIPLIER
  
Для СОГЛАСНЫХ (защита):
  shield = base_power × letter_level × CONSONANT_MULTIPLIER
  
Для Ъ (твёрдый знак — бафф защиты):
  defense_bonus = base_power × letter_level × SIGN_MULTIPLIER
  (усиливает следующую согласную защиту)
  
Для Ь (мягкий знак — бафф атаки):
  attack_bonus = base_power × letter_level × SIGN_MULTIPLIER
  (усиливает следующую гласную атаку)
```

**Балансировочные константы** (настраиваются):
- `VOWEL_MULTIPLIER = 1.0` — множитель урона гласных
- `CONSONANT_MULTIPLIER = 1.0` — множитель защиты согласных
- `SIGN_MULTIPLIER = 1.5` — множитель для Ъ и Ь (они редкие, поэтому сильнее)

**Порядок хода**: В бою все участники выкладывают по 1 букве. Ходы разрешаются по скорости: от самой быстрой (33) к самой медленной (1). Быстрые бьют первыми, медленные — последними, но сильнее.

**HP**: У каждого участника боя есть HP. Когда HP ≤ 0 — бой проигран.

### 2.4 Стартовое состояние игрока

Игрок начинает игру **без единой буквы**. Вообще пустой инвентарь.

На старте можно найти только:
- Точки `.` (ресурс, лежат на земле в Светлой Долине)
- Первые буквы (спрятанные, как награды за исследование)

## 3. ЗНАКИ ПРЕПИНАНИЯ — ОСОБЫЕ КАРТЫ

Знаки препинания — это особые способности, не привязанные к алфавитному порядку.

| Знак | Название | Эффект | Где найти |
|------|----------|--------|-----------|
| `.` | Точка | Базовый ресурс. Собираешь 3 точки → комбинируются в `...` | Повсюду в Светлой Долине, реже дальше |
| `...` | Многоточие | **Разговор с монстрами/NPC**. Позволяет избежать боя, получить информацию, торговать, убеждать. Ключевая механика старта. | Комбинируется из 3 точек |
| `,` | Запятая | **Пауза** — замедляет врага на 1 ход (его скорость −5) | Леса, болота |
| `?` | Вопрос | **Подозревает** — монстр использует: патрулирует, замечает движение игрока. Стартовый монстр | Только монстры (не игрок, поначалу) |
| `!` | Восклицание | **Заметил** — монстр переходит в агрессию, ускоряется, атакует. Стартовый монстр | Только монстры (не игрок, поначалу) |
| `-` | Дефис | **Уклонение/рывок** — позволяет уклониться от одной атаки | Катакомбы |
| `:` | Двоеточие | **Раскрытие** — показывает скрытые карты/намерения врага на следующий ход | Цитадели |
| `;` | Точка с запятой | **Комбо-связка** — позволяет сыграть 2 буквы за 1 ход вместо одной | Редкий, руины |
| `""` | Кавычки | **Отражение** — возвращает следующую вражескую атаку обратно | Очень редкий |
| `—` | Тире | **Прорыв** — пробивает вражескую защиту, игнорирует щит | Боссы |
| `()` | Скобки | **Ловушка** — ставится на поле, срабатывает при атаке врага (урон при атаке на тебя) | Болота |
| `...` + буква | Убеждение | Комбо: многоточие + конкретная буква = убеждение монстра определённым «словом» (расширенная дипломатия) | Продвинутая механика |

**Важно**: На старте игрок может ТОЛЬКО собирать точки и использовать многоточие. Никакого оружия, никаких атак. Единственный способ взаимодействовать с монстрами — говорить через `...`.

## 4. КАРТА МИРА

### 4.1 Общая структура

- **Единая огромная карта** без разделения на уровни/комнаты
- **33 региона** на карте (по числу букв, но регионы НЕ привязаны к конкретным буквам — в каждом регионе можно найти разные буквы)
- Реализуется через **TileMap / TileMapLayer** в Godot
- Seamless — единая карта, камера следует за игроком

### 4.2 Визуальный стиль

- **Мрачное средневековье**: тёмные тона, приглушённые цвета
- **Контраст**: стартовая область — светлая, залитая солнцем, уютная
- По мере удаления от старта — мир темнеет, становится опаснее
- **Арт-стиль**: Полигональный / low-poly (для производительности на web и mobile)
- **Источники ассетов**: PolyHaven (бесплатные текстуры/модели), ручная генерация

## 5. СИСТЕМА МОНЫСТРОВ

### 5.1 Стартовые монстры (Светлая Долина)

**Монстр «?» (Вопросительный)**:
- Патрулирует территорию
- Активность: «подозревает» — если игрок рядом, начинает искать
- Радиус обнаружения: малый
- При обнаружении → переходит в состояние «!»
- HP: низкий, но без оружия убить нельзя
- Убеждение через `...`: «Ты не знаешь, кто я. Уходи.» → монстр уходит

**Монстр «!» (Восклицательный)**:
- Агрессивный, атакует при контакте
- Быстрый, но слабый
- HP: средний
- Без оружия — только убеждение через `...`
- Убеждение: «Подожди! Я не враг!» → монстр замедляется, можно убежать

### 5.2 Прогрессия монстров

По мере продвижения вглубь карты:
- Монстры начинают использовать буквы (атакуют гласными, защищаются согласными)
- Появляются монстры с комбинациями: `?` + буква = хитрый монстр, `!` + буква = агрессивный
- Боссы регионов: уникальные монстры с набором букв и знаков
- **Финальный босс**: Хранитель Запрета — владеет полным алфавитом

### 5.3 Дроп с монстров

- После убеждения (`...`): монстр может отдать букву, точку, информацию
- После победы в бою: монстр роняет 1-3 буквы из своего набора
- Редкие монстры: роняют редкие знаки препинания

## 6. СЮЖЕТ И НАРРАТИВ

### 6.1 Лор

Когда-то в мире звучала речь. Люди писали книги, называли вещи по имени, пели песни. Буквы жили в каждом доме, в каждом слове.

Но пришёл **Хранитель Запрета** — древняя сущность, рождённая из страха перед силой слова. Он наложил проклятье: буквы были разбросаны по тёмным землям, за пределы Светлой Долины. Способность говорить и писать была утрачена. Даже в Светлой Долине — последнем оплоте — буквы запрещены под страхом изгнания.

Но нашлись герои. Те, кто решил отправиться в тёмные земли и вернуть буквы одну за одной. Восстановить алфавит. Разрушить проклятье.

### 6.2 Ключевые персонажи/NPC

- **Староста деревни** (Светлая Долина) — даёт первые задания, учит механикам
- **Тихий Странник** — NPC в лесу, обменивает точки на подсказки
- **Библиотекарь Призрак** — в руинах библиотеки, знает где спрятаны буквы
- **Кузнец Слов** — в цитадели, может улучшать (комбинировать) буквы
- **Хранитель Запрета** — финальный босс, владеет всеми 33 буквами

## 7. МУЛЬТИПЛЕЕР

### 7.1 Архитектура

- **Godot SceneMultiplayer** (высокоуровневый API)
- **ENet** для PC нативных клиентов
- **WebSocket** для web-клиентов (itch.io не поддерживает UDP/ENet)
- **Сервер**: выделенный Godot headless сервер или peer-to-peer с хостом
- **WebRTC** для голосового чата (отдельный сигнальный сервер)

### 7.2 Функционал

- **Лобби**: Создать/найти комнату
- **Текстовый чат**: глобальный + локальный (в регионе)
- **Голосовой чат**: Push-to-talk или активация по голосу, внутри группы
- **Торговля**:直接的 обмен буквами между игроками (торговое окно)
- **PvP**: Дуэли на буквах (арена или в мире)
- **Кооп**: Совместное исследование мира, групповые бои с боссами
- **Синхронизация мира**: Позиции игроков, состояние монстров, ресурсы

### 7.3 Приоритет реализации

Мультиплеер реализуется ПОСЛЕ основной одиночной игры. Сначала работающая одиночка, потом сетка.

## 10. КОНВЕНЦИИ КОДА

### 10.1 Именование

- **Файлы**: `snake_case.gd` (например, `combat_system.gd`)
- **Классы/ноды**: `PascalCase` (например, `CombatSystem`)
- **Переменные**: `snake_case` (например, `base_power`)
- **Константы**: `UPPER_SNAKE_CASE` (например, `VOWEL_MULTIPLIER`)
- **Функции**: `snake_case` (например, `calculate_damage()`)

### 10.2 Структура GDScript-файлов

```gdscript
extends Node
class_name ClassName

# Константы
const EXAMPLE_CONST = 10

# Экспортируемые переменные
@export var example_export: int = 5

# Переменные
var example_var: int = 0

# Сигналы
signal example_signal(value: int)

# Встроенные функции (_ready, _process и т.д.)
func _ready() -> void:
    pass

# Публичные функции
func public_function() -> void:
    pass

# Приватные функции
func _private_helper() -> void:
    pass
```

### 10.3 Правила

- Типизировать всё явно (`var x: int = 0`, не `var x = 0`)
- Не использовать `@onready` для Autoload-синглтонов (они уже готовы)
- Предпочитать Composition over Inheritance
- Комментарии — только для сложной логики, не для очевидного
- Каждый скрипт — одна ответственность

### 12.4 Рабочий цикл AI-агента (SELF-DRIVEN DEVELOPMENT)

Агент работает **самостоятельно**, видит игру глазами и итерирует без участия человека:

1. **Реализовать/изменить фичу** → написать код
2. **Собрать Godot HTML5 build** → запустить на `http://localhost:3000`
3. **Пройти игру через Puppeteer** → сделать скриншоты + собрать логи
4. **Проанализировать скриншоты через Vision MCP** → увидеть баги глазами
5. **Починить баги** → повторить цикл
6. **Запустить regression тесты** → убедиться что ничего не сломалось
7. **Запустить full playthrough** → убедиться что игра проходится

Агент **САМ** эмулирует шаги игрока, делает скриншоты, анализирует их через vision, и итерирует до тех пор пока всё не работает.

### 12.5 Vision MCP Server — глаза AI-агента

Vision-анализ доступен **ДВУМЯ способами** (использовать оба, fallback друг на друга):

#### Способ 1 — встроенные инструменты opencode (in-process)

В `opencode.json` подключён `zai-mcp-server` с `Z_AI_API_KEY`. Доступно напрямую как вызовы функций:
- `zai-mcp-server_analyze_image` — общий анализ изображения
- `zai-mcp-server_extract_text_from_screenshot` — OCR (распознавание текста)
- `zai-mcp-server_diagnose_error_screenshot` — диагностика ошибок
- `zai-mcp-server_ui_diff_check` — сравнение скриншотов (до/после)
- `zai-mcp-server_understand_technical_diagram` — анализ диаграмм
- `zai-mcp-server_analyze_data_visualization` — анализ графиков
- `zai-mcp-server_ui_to_artifact` — UI → код/спецификация
- `zai-mcp-server_analyze_video` — анализ видео (MP4/MOV/M4V, ≤8MB)

Параметры: `image_source` (локальный путь или URL) + `prompt`.

#### Способ 2 — внешний `mcp-cli.exe` (fallback, работает из любого проекта)

- Бинарник: `~\.bun\bin\mcp-cli.exe` (Bun, mcp-cli v0.3.0)
- Конфиг: `~\.mcp_servers.json` (сервер `zai-vision`, модель GLM-4.6V)
- Работает из любой директории

**Перед каждым вызовом обновить PATH:**
```powershell
$env:Path = [System.Environment]::GetEnvironmentVariable("Path","User") + ";" + [System.Environment]::GetEnvironmentVariable("Path","Machine")
```

**Команды:**
```powershell
# Анализ изображения
mcp-cli call zai-vision analyze_image '{"image_source":"C:/path/file.png","prompt":"Опиши что видишь"}'

# OCR — распознавание текста
mcp-cli call zai-vision extract_text_from_screenshot '{"image_source":"C:/path/file.png","prompt":"Извлеки текст"}'

# Сравнение двух скриншотов
mcp-cli call zai-vision ui_diff_check '{"expected_image_source":"ref.png","actual_image_source":"cur.png","prompt":"Сравни"}'

# Диагностика ошибки
mcp-cli call zai-vision diagnose_error_screenshot '{"image_source":"err.png","prompt":"Что за ошибка"}'

# Список серверов и инструментов
mcp-cli -d
```

**Правила:**
- JSON-аргументы в одинарных кавычках: `'{"key":"value"}'`
- Пути с `/` (forward slash), не `\`
- Каждый вызов ≈ 10–15 секунд
- Сначала пробовать встроенные инструменты, потом mcp-cli как fallback

**Обёртка с таймаутом:** `scripts/dev/vision.ps1 -Tool <name> -Json '<json>' -TimeoutSec 90` — убивает mcp-cli при зависании (см. §12.8).

**Как использовать в рабочем цикле:**
1. Сделать скриншот: `gameActions.snapshot('bug_report')` или `godot.takeScreenshot()`
2. Скриншоты сохраняются в `tests/e2e/screenshots/`
3. Передать скриншот Vision MCP (способ 1 или 2)
4. MCP проанализирует и вернёт описание/диагноз
5. Агент чинит на основе визуального анализа

### 12.6 Система скриншотов (ScreenshotManager)

### 12.8 Правило таймаутов и авто-восстановления

**ЛЮБАЯ** команда (build / serve / test / vision / godot), которая может зависнуть, **ОБЯЗАНА** запускаться с таймаутом. Если процесс не уложился — убивать и сообщать где зависло, чтобы агент мог восстановиться в автоматическом режиме.

**Dev-скрипты** (в `scripts/dev/`):

| Скрипт | Назначение | Таймаут по умолчанию |
|--------|------------|----------------------|
| `build.ps1` | HTML5 export через Godot headless. Kill если зависнет | 300с |
| `serve.ps1` | http-server на :3000 с авто-рестартом при падении (≤20 рестартов) | watchdog 5с poll |
| `test.ps1` | jest e2e с per-test timeout, авто-rerun flake, общий timeout | 1500с overall, 120с/тест |
| `vision.ps1` | Обёртка над `mcp-cli call zai-vision ...` | 90с |
| `verify.ps1` | Сборка→сервер→smoke→vision главной страницы | 300с |

**Примеры:**
```powershell
# Сборка с таймаутом
& scripts/dev/build.ps1 -TimeoutSec 300

# Тесты с авто-flake-rerun
& scripts/dev/test.ps1 -Suite "user/" -OverallTimeoutSec 1500

# Vision анализ скриншота
& scripts/dev/vision.ps1 -Tool analyze_image -Json '{"image_source":"C:/path/s.png","prompt":"Опиши"}' -TimeoutSec 90
```

**Принципы авто-восстановления:**
1. Каждый шаг логирует начало/конец/длительность
2. При timeout — kill + exit code 124 + продолжить следующий шаг
3. При краше сервера — авто-рестарт (≤20 раз), потом fail
4. При flake теста — один rerun, потом accept fail
5. Никаких `Start-Process ... -Wait` без `WaitForExit($ms)` — только параметризованные таймауты
6. Все логи в `$env:TEMP\bookwar_*.log` — агент может их прочитать

## 13. ТЕСТИРОВАНИЕ

### 13.1 Стратегия тестирования

Тестирование проводится **не вызовом внутренних функций напрямую**, а через **пользовательские (user) и компонентные (component) тесты**, моделирующие реальные действия игрока. Каждое тестовое действие проходит через тот же путь, что и реальный пользовательский ввод.

**КЛЮЧЕВОЕ ПРАВИЛО**: Никогда не вызывать внутренние GDScript функции из тестов. Тесты взаимодействуют с игрой только через Puppeteer — клавиатура, мышь, чтение `window.*` переменных.

### 13.2 E2E тесты через Puppeteer

Поскольку целевая платформа — itch.io (HTML5 export), e2e тестирование проводится через **Puppeteer**, запускающий Godot HTML5 build в headless Chromium.

**URL игры**: `http://localhost:3000`

**ВАЖНО — Puppeteer на Windows**: Chrome запускается с флагом `--disable-features=ntlm-auth`, чтобы отключить Windows-аутентификацию. Без этого флага учётная запись пользователя блокируется. Флаг уже прописан в `godot_page.js`.

**Структура тестов:**
```
tests/e2e/
├── package.json              ← Puppeteer + Jest
├── jest.config.js            ← Jest config (timeout 120s, port 3000)
├── screenshots/              ← Скриншоты из тестов + для Vision MCP
├── helpers/
│   ├── godot_page.js         ← Puppeteer: браузер, скриншоты, JS bridge, debugCapture
│   └── game_actions.js       ← Высокоуровневые действия (двигаться, подобрать, атаковать)
├── user/                     ← Пользовательские тесты (действия игрока)
│   ├── startup.test.js       ← Загрузка, меню, новая игра

│   ├── pickup.test.js        ← Подбор точек/букв
│   ├── inventory.test.js     ← Открытие/закрытие инвентаря
│   ├── dialogue.test.js      ← Диалог через многоточие
│   └── exploration.test.js   ← Исследование Светлой Долины
├── component/                ← Компонентные тесты (подсистемы через UI)
```

### 13.3 Принципы пользовательских тестов (User Tests)

Пользовательские тесты моделируют действия реального игрока:
- **Не вызывать внутренние функции** напрямую (нет `combat_system.calculate_damage()`)
- **Действовать через UI/ввод**: кликнуть кнопку, нажать клавишу, переместить мышь
- **Проверять видимый результат**: что на экране отображается ожидаемый результат
- Каждый тест описывает сценарий с точки зрения игрока

Пример:
```javascript
// ПРАВИЛЬНО — пользовательский тест
test('игрок подбирает точку и видит её в инвентаре', async () => {
  await gameActions.movePlayerTo(page, dotPosition);
  await gameActions.interact();
  const inventory = await gameActions.getInventoryContents();
  expect(inventory.dots).toBeGreaterThan(0);
});

// НЕПРАВИЛЬНО — вызов внутренней функции
test('inventory adds dot', () => {
  const inv = new Inventory();
  inv.add_item('dot');
  expect(inv.has_item('dot')).toBe(true);
});
```

### 13.4 Принципы компонентных тестов (Component Tests)

Компонентные тесты проверяют изолированные подсистемы через их **внешний интерфейс**:
- Боевая система: инициировать бой через UI → проверить результат на экране
- Инвентарь: открыть через UI → проверить отображение
- Монстры: подойти к монстру → проверить реакцию (агрессия, диалог)
- HUD: проверить что данные отображаются корректно

### 13.5 Regression-тесты (REGRESSION GUARD)

**Цель regression-тестов**: убедиться что починенное не ломается снова.

После каждого изменения кода агент запускает:
```
cd tests/e2e && npm run test:component
```

Regression-тесты включены в `component/`:
- `regression_smoke.test.js` — загрузка, движение, инвентарь, нет ошибок
- `regression_data.test.js` — 33 буквы загружены, пустой старт, данные валидны
- `regression_combat.test.js` — бой корректно стартует/завершается
- `regression_ui.test.js` — HUD, инвентарь, скриншоты не падают

**Правило**: Если баг был починен, добавь regression-тест который проверяет именно этот кейс.

### 13.6 Система скриншотов в тестах

Каждый тест может делать скриншоты:

```javascript
// Полный скриншот страницы
await godot.takeScreenshot('имя');

// Скриншот только canvas (игры)
await godot.takeCanvasScreenshot('имя');

// Скриншот + состояние + логи — всё разом
const debug = await godot.debugCapture('bug_report');
// debug.state — весь game state
// debug.screenshot — путь к файлу скриншота
// debug.canvas — путь к canvas скриншоту
// debug.consoleLog — логи консоли с ошибками
```

Скриншоты сохраняются в `tests/e2e/screenshots/`. Vision MCP анализирует эти файлы.

### 13.7 Полное прохождение (Playthrough Test)

`tests/e2e/playthrough/full_run.test.js` — главный тест. Проходит весь стартовый контент:
1. Запуск игры → главное меню
2. Новая игра → спавн в Светлой Долине
3. Движение по карте → сбор точек
4. Комбинация 3 точек → многоточие
5. Подход к монстру ? → диалог через многоточие → монстр уходит
6. Подбор первой буквы → проверка инвентаря
7. Первый бой с монстром ! → победа → проверка дропа
8. Возврат в деревню → диалог со старостой

### 13.8 Инструменты

- **Puppeteer** — управление headless Chromium с Godot HTML5 build
- **Jest** — test runner (timeout 120s)
- **godot_page.js** — хелперы: браузер, скриншоты, JS bridge, debugCapture, логи
- **game_actions.js** — высокоуровневые действия: движение, инвентарь, диалог
- **Vision MCP** — анализ скриншотов через Z.AI GLM-4.6V
- **ScreenshotManager** — внутриигровая система скриншотов (GDScript)

### 13.9 Запуск тестов

```bash
cd tests/e2e
npm test                        # Все тесты
npm run test:user               # Только пользовательские
npm run test:component          # Только компонентные + regression
npm run test:playthrough        # Только полное прохождение
```

### 13.10 Правила тестирования для AI-агентов

- ВСЕГДА пиши пользовательские тесты, не вызывающие внутренние функции
- КАЖДАЯ фича сопровождается компонентным тестом
- КАЖДЫЙ починенный баг сопровождается regression-тестом
- ПОЛНОЕ ПРОХОЖДЕНИЕ должно проходить зелёно перед завершением работы
- ТЕСТИРУЙ через Godot HTML5 build + Puppeteer, не через GDScript unit tests
- ПРИ ИЗМЕНЕНИИ логики — запускай regression тесты
- ДЕЛАЙ скриншоты при ошибках — `godot.debugCapture('error_name')`
- АНАЛИЗИРУЙ скриншоты через Vision MCP — не угадывай, а смотри
- Puppeteer ВСЕГДА запускается с `--disable-features=ntlm-auth`

## 15. ДОПОЛНИТЕЛЬНЫЕ ЗАМЕТКИ

### 15.1 Баланс (предварительный)

- Медленные буквы (А, Б) — мощные, но ходят последними. Риск: получишь урон до хода.
- Быстрые буквы (Э, Ю, Я) — слабые, но ходят первыми. Хорошо для добивания.
- Оптимальная стратегия: баланс скорости и силы в колоде.
- Ъ + согласная = усиленная защита (комбо из 2 карт)
- Ь + гласная = усиленная атака (комбо из 2 карт)

### 15.2 Идеи на будущее

- **Составление слов**: Собрать буквы в слово → бонусный эффект (например, «ЩИТ» = двойная защита)
- **Английский алфавит**: Второй набор букв для расширения
- **Рейтинг игроков**: На основе собранного алфавита
- **Ивенты**: Временные события с редкими буквами
- **Крафт**: Комбинация букв для создания новых знаков препинания
